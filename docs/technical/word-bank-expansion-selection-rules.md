# 單字庫擴充挑字規則 (Word Bank Expansion Selection Rules)

本文記錄從 `assets/word_bank/en` 持續擴充 `assets/word_bank/word_bank_main.json` 時，實際採用的挑字與批次處理規則。

## 1. 目的

這套流程的目標不是單純把 `assets/word_bank/en` 依序灌進主庫，而是：

- 優先加入**常用、可教學、可解釋**的英文詞條。
- 避免把**噪音、專有名詞、明顯變化形、冷僻字、敏感詞**大量寫入主庫。
- 每次新增時都保持 `word_bank_main.json` **可排序、可驗證、可持續擴充**。

## 2. 資料來源

- 主來源：`assets/word_bank/en`
- 目標檔案：`assets/word_bank/word_bank_main.json`

`assets/word_bank/en` 是一份原始英文單字清單，不是已整理好的教學詞表，因此不能直接按行匯入。

## 3. 挑字流程總覽

整體流程如下：

1. 從 `assets/word_bank/en` 讀出所有候選字。
2. 先排除不適合直接收錄的項目。
3. 對剩餘詞條做常用度排序。
4. 逐批查詞典，確認詞性、定義、例句是否可用。
5. 生成符合主庫 schema 的資料。
6. 對詞典定義與例句先做純文字清理（sanitize）。
7. 每累積 40 筆就寫回 `word_bank_main.json`。
8. 每次寫回後立即依 `word` 重新排序整份主庫。
9. 寫回前後都用 validator 檢查主庫。
10. 重複上述步驟，直到達到目標筆數。

## 4. 候選字第一層過濾

從 `assets/word_bank/en` 讀出候選字後，會先做基本過濾：

- 只保留全小寫英文字母詞條：`^[a-z]+$`
- 排除長度過短或過長的字
  - 目前實作大致落在 `4` 到 `12` 個字母
- 排除已存在 `word_bank_main.json` 的字
- 排除和現有詞條高度重疊的 token
  - 例如主庫已有 `actor/actress`，就不一定再接受完全重複的拆分變體
- 排除明顯變化形
  - 例如結尾是 `s`、`ed`、`ing` 的項目
- 排除黑名單詞條
  - 包含部分專有名詞、地名、人名、髒話、明顯不適合作為一般單字庫的詞

注意：這一層只是初篩，不代表留下來的字就一定會進主庫。

## 5. 候選字排序規則

挑字順序**不是按照字母順序**。

實際上會先用詞頻工具做常用度排序：

- 詞頻高的字優先處理
- 詞頻過低的字直接淘汰

目前的實作思路是：

- 先用 `wordfreq` 計算英文詞頻
- 只保留高於最低門檻的詞
- 再按「詞頻由高到低」排序

因此，寫回主庫時雖然最後會重新依字母排序，但**被挑進處理流程的順序**不是字母順序，而是「過濾後的常用度優先順序」。

## 6. 詞典驗證規則

每個候選字不會直接寫入主庫，而是先查詞典資料。

目前主要使用可程式化取得的英文字典/定義來源，目的是取得：

- `partOfSpeech`
- 英文定義
- 可用的英文例句

如果詞典資料出現以下情況，通常會排除：

- 只是變化形說明
  - 例如 `plural of`、`past tense of`、`present participle of`
- 只是替代拼法或舊拼法
  - 例如 `alternative spelling of`、`obsolete form of`
- 明顯是專有名詞、人名、姓氏、地名
- 定義帶有明顯敏感或不適合作為主庫教學詞條的標記
- 定義品質太差、太短、太亂、含雜訊

## 7. 主庫資料生成規則

通過詞典驗證後，才會生成主庫所需資料格式：

```json
{
  "word": "example",
  "meaning": "中文意思。",
  "partOfSpeech": "noun",
  "sentences": [
    "English sentence. 中文翻譯。",
    "English sentence. 中文翻譯。"
  ]
}
```

生成時遵守以下原則：

- `word`：單一 canonical 詞條
- `meaning`：以繁體中文表達，盡量可讀
- `partOfSpeech`：依詞典資料映射到主庫可接受的詞性
- `sentences`：
  - 固定 2 句
  - 優先使用詞典原例句
  - 若詞典沒有例句，才使用 fallback 模板句
  - 句子最終會轉成單行格式
  - 不允許保留任何 HTML / wiki 標記

補充：若詞典來源回傳的 `definition` 或 `example` 含有 `<a ...>`、`<span ...>`、`<b>...</b>` 等標記，必須先清理成純文字後，才能進入翻譯與寫回流程。不能把這些內容直接當作 plain text 存回主庫。

## 8. 寫回批次規則

這裡要區分兩種批次概念：

- **候選視窗大小**
  - 一次先抓一批較大的候選字來查詞典
  - 目的是提升處理效率
- **寫回批次大小**
  - 每累積 `40` 筆合格單字，就寫回一次主庫

因此：

- `40` 是**寫回大小**
- `1000` 或 `2000` 是**總目標數量**

不是把 `assets/word_bank/en` 按字母順序每 40 行直接寫進去。

## 9. 排序規則

每次寫回後，都會對整份 `assets/word_bank/word_bank_main.json` 重新排序：

- 依 `word.casefold()` 做不分大小寫排序
- 確保主庫維持可讀、可搜尋、可比較的順序

所以：

- 挑字順序不是字母順序
- 但**最終主庫一定會是字母順序**

## 10. 驗證規則

寫回前後都應執行主庫驗證。至少要檢查：

- JSON 可正常解析
- `word` 不重複
- 整份主庫依 `word.casefold()` 排序
- 每筆都有 `word`、`meaning`、`partOfSpeech`、`sentences`
- `partOfSpeech` 必須在允許集合內
- `sentences` 必須固定為 2 句
- `meaning` 與 `sentences` 不允許含有 HTML / wiki 標記
- `meaning` 與 `sentences` 不允許含有換行

目前對應工具：

- `python3 tools/validate_word_bank_main.py`
- `python3 tools/validate_word_bank_main.py --fail-on-warnings`

建議用法：

- 日常新增後檢查：`python3 tools/validate_word_bank_main.py`
  - 只要沒有 `error` 就可視為結構安全
  - 舊資料造成的 newline / 舊詞性格式會列為 `warning`
- 嚴格清理模式：`python3 tools/validate_word_bank_main.py --fail-on-warnings`
  - 當要清歷史資料時再使用

## 11. 停止條件

常見停止條件有兩種：

1. 已達到目標數量
   - 例如新增滿 `1000` 筆
2. 候選字品質明顯下降
   - 開始大量出現冷僻詞、技術詞、專名、敏感詞、錯義項

當第二種情況出現時，應優先停止並交由人工 review，而不是繼續灌數量。

## 12. 品質下降判定規則

「候選池後段品質開始明顯下降」不是單一指標，而是多個訊號同時出現。實務上可用以下規則判定：

1. 候選接受率下降
   - 每個候選視窗（例如 300 個詞）中，最終能通過詞典驗證並寫入主庫的比例明顯下滑。
   - 若連續多個視窗都只能留下少量詞條，代表候選池正在惡化。

2. 教學價值下降
   - 日常、通用、可教學的字變少。
   - 開始出現更多冷僻字、過度書面字、宗教字、歷史字、專門術語、專名邊界詞。

3. 詞典義項品質下降
   - 抓到的定義不是主流義項。
   - 第一義是專業義、延伸義、少見義、地區義，或必須大量人工判斷才能決定是否收錄。

4. 例句品質下降
   - 可直接使用的自然例句變少。
   - fallback 例句比例提高。
   - 句子雖然合法，但越來越像模板句，生活化程度下降。

5. 中文自然度下降
   - 自動翻譯越來越容易出現直譯、僵硬表達或教材感不佳的中文。
   - 需要人工潤稿的比例提高。

6. 可疑詞比例提高
   - 專名、敏感詞、拼寫變體、低頻詞、俚語或黑名單邊界詞出現頻率提高。
   - 即使最後沒有寫入，也表示候選池後段噪音比例在增加。

可操作的簡化判定方式：

- 前段候選：找到字之後，多半值得收錄。
- 後段候選：找到字之後，必須先懷疑它是否真的值得收錄。

若流程中出現以下任一情況，建議停止本輪擴充並改成人工 review：

- 連續 2 到 3 個候選視窗接受率明顯下降
- fallback 例句比例明顯升高
- 可疑詞與低教學價值詞開始頻繁混入候選結果
- review 成本開始高於新增價值

## 13. 已知風險

這套流程能快速擴充主庫，但仍有明確限制：

- 詞典可能提供非主要義項
- 詞典 API 可能回傳帶標記的 definition/example，若未 sanitize 會污染主庫
- 自動翻譯可能偏直譯
- fallback 例句自然度不一定穩定
- 常用度高不代表一定適合教學
- 在候選池後段，偏門詞比例會上升

因此，建議把大量新增視為：

- **結構穩定的草稿資料**
- 而不是最終可直接上線的教學內容

## 14. 建議的實務節奏

目前較穩妥的節奏如下：

1. 每次新增 `600` 到 `1000` 筆
2. 寫回後立即排序
3. 做一次人工 review
4. 清掉不理想詞條後，再進下一輪

若一次拉到 `2000` 筆，風險通常不是 JSON 結構，而是：

- 候選字品質下滑
- review 成本大幅上升
- 不理想詞條混入主庫的比例變高

## 15. 結論

目前主庫擴充不是採用「依字母順序，每 40 筆處理一次，直到湊滿 1000」這種單純模式。

實際模式是：

- 從 `assets/word_bank/en` 抽候選字
- 做多層過濾
- 依常用度優先
- 用詞典再驗證一次
- 生成主庫格式資料
- 每 40 筆寫回一次
- 每次寫回後自動重新排序
- 持續重複到達目標數量或人工決定停止
