# 使用者自建單字回饋方案評估

- Date: 2026-03-10
- Status: Draft

## 1. 目標

希望達成兩件事：

1. 使用者可以在 App 內新增自己的單字。
2. 使用者若願意，可以把自己新增的單字提供給開發者，作為未來主字庫擴充或人工整理的來源。

第一階段以目前專案的 Flutter + 本地資料庫 + CloudKit 架構為前提，優先選擇低風險、低維運成本的做法。

## 2. 現況整理

目前專案其實已支援新增單字，但還不支援「回饋給開發者」：

1. 手動新增單字已存在。
   - `AddWordPage` 會呼叫 `WordsNotifier.addWord()` 建立 `WordCard`。
2. 內建字庫加入單字也走同一條儲存流程。
   - `WordBankPage` 內加入字庫單字時，同樣呼叫 `WordsNotifier.addWord()`。
3. `WordCard` 目前沒有資料來源欄位。
   - 系統無法分辨一筆資料是：
     - 使用者手動新增
     - 從內建字庫加入
     - 舊版本留下來、來源未知
4. 目前雲端同步使用 CloudKit Private Database。
   - 這只能支援同一位使用者自己的 iOS/macOS 裝置同步。
   - 不能直接當成「把所有使用者新增單字集中給開發者」的管道。

## 3. 核心限制

### 3.1 現有 CloudKit 不能直接拿來收集使用者資料

目前同步設計是 Private Database，這代表資料屬於使用者自己的 iCloud 私有空間，不是開發者可集中讀取的資料池。

### 3.2 既有資料無法可靠回推來源

因為目前 `WordCard` 沒有 `origin/source` 欄位，舊資料無法準確知道是手動新增還是從內建字庫加入。

這代表若要開始做「提供給開發者」功能，必須先接受一個事實：

- 新資料可以精確標記來源。
- 舊資料頂多標成 `unknown`，不應自動當成可提交內容。

### 3.3 圖片不適合列為第一階段提交內容

圖片會增加：

- 檔案大小
- 分享失敗率
- 隱私風險
- 後端儲存與審核成本

所以第一階段建議只提交文字資料，圖片最多只保留 `hasImage` 布林資訊，不上傳實際圖片。

## 4. 評估準則

衡量方案時，建議看這六個面向：

1. 開發成本
2. 維運成本
3. 使用者操作摩擦
4. 隱私與合規風險
5. 資料品質與可整理性
6. 後續擴充性

## 5. 方案比較

| 方案 | 說明 | 開發成本 | 維運成本 | 使用者體驗 | 資料品質 | 建議 |
| --- | --- | --- | --- | --- | --- | --- |
| A. 匯出檔案後手動分享 | 產生 JSON/CSV，讓使用者透過分享表單、Mail、AirDrop、訊息傳給你 | 低 | 低 | 中 | 高 | 建議先做 |
| B. App 內直接送到自有 API | 使用者按送出，App 直接上傳到你的後端 | 中到高 | 中到高 | 高 | 高 | 第二階段再做 |
| C. 改用 CloudKit Public/Shared Database 收集 | 讓 App 直接寫到可共享的 CloudKit 區域 | 高 | 中 | 中 | 中到低 | 不建議 |

### 方案 A：匯出檔案後手動分享

做法：

1. App 內提供「分享我新增的單字給開發者」入口。
2. 使用者先看到預覽與說明。
3. App 產出一個 JSON 檔。
4. 使用者自行選擇 Mail、AirDrop、訊息或存到 Files，再交給你。

優點：

1. 不需要新後端。
2. 最符合目前專案狀態。
3. 由使用者明確操作，隱私風險最低。
4. 失敗模式簡單，容易 debug。
5. 你可以先驗證是否真的有使用者會願意提交資料。

缺點：

1. 需要使用者多按幾步。
2. 開發者端仍要手動整理檔案。
3. 若只用 `mailto:`，會遇到 URL 長度限制，所以應以「分享檔案」為主，不建議只塞 email body。

### 方案 B：App 內直接送到自有 API

做法：

1. App 內做提交按鈕。
2. 後端接收批次 JSON。
3. 後端做去重、排隊、審核與後續整理。

優點：

1. 使用者流程最短。
2. 開發者端資料集中。
3. 可以做版本控管、審核狀態、統計分析。

缺點：

1. 需要新後端、資料儲存與監控。
2. 需要處理隱私揭露與資料收集說明。
3. 需要防濫用、重送、重複提交、格式錯誤等問題。

### 方案 C：改用 CloudKit Public/Shared Database 收集

理論上可做，但對這個需求不划算：

1. 會讓現在簡單清楚的 Private Database 架構變複雜。
2. 不利於做提交審核與內容治理。
3. 實作與測試成本高，收益不如直接做匯出或 API。

結論：不建議把這個需求建立在 CloudKit Public/Shared Database 上。

## 6. 推薦路線

推薦採用：

1. 第一階段：方案 A
2. 第二階段：若提交量真的有成長，再升級到方案 B

原因：

1. 現在最缺的是「可辨識來源」與「可交付格式」，不是大型同步基礎設施。
2. 先做匯出分享，可以最快驗證真實需求。
3. 一旦提交量很低，你就不必背負後端維運成本。
4. 一旦提交量很高，先前定義好的 JSON schema 與來源欄位可以直接沿用到 API。

## 7. 建議的資料設計

### 7.1 在 `WordCard` 新增來源欄位

建議新增：

```text
enum WordOrigin {
  unknown,
  manual,
  builtinWordBank,
}
```

規則：

1. `AddWordPage` 新增的單字標成 `manual`
2. `WordBankPage` 從主字庫加入的單字標成 `builtinWordBank`
3. 舊資料 migration 預設標成 `unknown`

### 7.2 在 `WordCard` 新增開發者分享狀態

建議新增：

```text
enum DeveloperShareState {
  notShared,
  exported,
  submitted,
}
```

最小欄位可先做：

- `developerShareState`
- `developerSharedAt`

如果只做方案 A，其實 `submitted` 可晚點再加；第一階段至少要有 `notShared/exported`，避免重複匯出難以追蹤。

### 7.3 匯出 payload 不要直接等同 `WordCard`

建議另外定義匯出格式，例如：

```json
{
  "schemaVersion": 1,
  "appVersion": "1.0.1+2",
  "platform": "ios",
  "exportedAt": "2026-03-10T12:00:00Z",
  "words": [
    {
      "id": "uuid",
      "origin": "manual",
      "word": "inspiration",
      "meaning": "靈感",
      "partOfSpeech": "noun",
      "sentences": ["Sentence 1", "Sentence 2"],
      "createdAt": "2026-03-10T11:50:00Z",
      "updatedAt": "2026-03-10T11:50:00Z",
      "hasImage": false
    }
  ]
}
```

這樣的好處是：

1. 可和本地儲存 schema 解耦
2. 未來改成本地 API 提交時，不需要重新設計 payload
3. 容易寫開發者端匯入工具

## 8. 分階段實作計劃

### Phase 1：補齊來源辨識

目標：先讓系統能分辨哪些是使用者自建內容。

工作項目：

1. 在 `WordCard`、local repository、CloudKit sync map 補 `origin`
2. 補 migration：
   - 舊資料預設 `unknown`
3. 修改新增流程：
   - `AddWordPage` -> `manual`
   - `WordBankPage` -> `builtinWordBank`
4. 在列表或設定頁加上可篩選或統計：
   - `manual`
   - `unknown`

驗收：

1. 新增手動單字後，資料來源正確
2. 從內建字庫加入後，資料來源正確
3. 舊資料不會被錯誤標成 `manual`

### Phase 2：匯出與分享

目標：讓使用者能明確把自建單字交給開發者。

工作項目：

1. 在 `Settings` 或 `Records` 新增入口：
   - `分享我新增的單字給開發者`
2. 提供提交前說明：
   - 只會分享文字資料
   - 不包含圖片檔本體
   - 會包含建立時間與更新時間
3. 允許選擇範圍：
   - 只分享 `manual`
   - 是否包含 `unknown`
   - 只分享尚未匯出的項目
4. 產出 JSON 檔
5. 透過分享功能送出
6. 成功後將資料標為 `exported`

驗收：

1. 使用者能看到待分享筆數
2. 匯出的 JSON 可被外部工具正常解析
3. 匯出後狀態正確更新

### Phase 3：開發者端整理工具

目標：降低你手動整理使用者回饋單字的成本。

建議新增一支工具腳本，例如：

- `tools/import_user_contributions.py`

功能：

1. 讀取使用者匯出的 JSON
2. 檢查 schema version
3. 依 `word.casefold()` 去重
4. 排除已存在於 `word_bank_main.json` 的詞
5. 把新詞輸出成待審核清單
6. 必要時轉成主字庫候選格式

驗收：

1. 你可以把使用者提供的檔案餵進工具
2. 工具能產出可 review 的候選清單
3. 與現有 `validate_word_bank_main.py` 流程可銜接

### Phase 4：視需求升級為直接提交 API

只有在以下情況出現時再做：

1. 使用者提交量穩定成長
2. 你不想再手動收檔案
3. 你需要後台統計、審核狀態與批次管理

此時再引入：

1. 簡單 API
2. 伺服器端資料庫
3. moderation queue
4. 重送與去重機制

## 9. 產品與 UX 建議

### 9.1 不要預設自動上傳

建議一定要是明確 opt-in：

1. 使用者知道自己正在分享什麼
2. 使用者自己按送出
3. 最好可先預覽

### 9.2 先只分享文字，不分享圖片

推薦第一階段分享欄位：

1. `word`
2. `meaning`
3. `partOfSpeech`
4. `sentences`
5. `origin`
6. `createdAt`
7. `updatedAt`
8. `hasImage`

不推薦第一階段分享：

1. 圖片 bytes
2. 本地圖片路徑
3. 完整複習歷史 `history`
4. 裝置可識別個資

### 9.3 UI 位置建議

建議把入口放在 `Settings`，其次才是 `Records`。

原因：

1. 這比較像資料分享/回饋設定，不是日常複習動作。
2. 使用者會預期在設定頁找到「聯絡開發者」或「分享資料」。

## 10. 風險與對策

### 10.1 舊資料來源不明

風險：

- 你可能會收到不是純手動新增的資料。

對策：

1. 將既有資料標成 `unknown`
2. 匯出 UI 預設只勾 `manual`
3. `unknown` 需要使用者額外確認才納入

### 10.2 使用者提交低品質內容

風險：

- 可能出現空白、重複、只有單字沒有意思、例句很差的內容。

對策：

1. 匯出前做最基本檢查
2. 開發者端 import 工具再做第二層清洗
3. 不直接自動寫入主字庫

### 10.3 若未來改成自動提交，會增加隱私與商店揭露壓力

對策：

1. 先用手動分享驗證需求
2. 真要做 API 時，再補資料收集說明與提交同意文字

## 11. 建議決策

如果現在要開始做，建議順序如下：

1. 先做 `origin` 欄位與 migration
2. 再做 `manual` 單字匯出 JSON
3. 再做分享入口與匯出狀態
4. 最後補開發者端整理工具

這條路徑的優點是：

1. 可以最快上線
2. 幾乎不增加維運負擔
3. 之後若升級成 API，也不會推翻前面的資料設計

## 12. 結論

目前最合理的做法不是改 CloudKit，而是：

1. 先補「單字來源標記」
2. 先做「使用者手動匯出並分享給開發者」
3. 等你確認真的有穩定貢獻流量後，再考慮做直接提交 API

這樣可以用最低成本，先把需求做成真正可用、可驗證、可持續擴充的功能。
