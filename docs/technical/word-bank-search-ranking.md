# Word Bank Search Ranking

## 背景

字庫搜尋目前是先依使用者輸入做 `contains` 過濾，再直接沿用字庫原始順序顯示結果。

由於內建字庫是依字母分 shard 載入後再合併，所以搜尋結果會偏向先顯示較前面字母的命中項，而不是最符合使用者意圖的單字。

例如搜尋 `money` 時，若 `baldmoney`、`commoney` 先出現在字庫中，就可能排在 `money` 前面。

## 問題

現行行為雖然能找到目標單字，但排序不符合搜尋直覺：

- 完全相同的單字沒有被優先顯示
- 字首匹配與單字內包含沒有區分
- 中文解釋命中與英文單字命中被混在同一層

這會讓使用者在字庫量變大後，越來越難快速找到最想加入複習庫的單字。

## 目標

在不改變空搜尋狀態與現有篩選功能的前提下，為字庫搜尋加入「匹配品質排序」，讓最可能是使用者目標的結果排在最前面。

## 排序規則

搜尋關鍵字存在時，依下列優先級排列：

1. 單字完全相同
2. 單字字首匹配
3. 單字內包含
4. 中文或解釋文字包含

同一優先級內，再用單字字母順序做 tie-breaker，確保結果穩定。

空搜尋時維持目前行為，仍只顯示前 100 筆，不做額外排序改動。

## 實作方式

新增一個 `WordBankSearchService`，將搜尋與排序邏輯從頁面元件中抽離。

Service 職責：

- 正規化 query
- 逐筆判斷 entry 的匹配等級
- 依等級分 bucket
- 對各 bucket 進行穩定排序
- 回傳限制筆數後的結果

頁面只保留：

- 讀取字庫
- 套用程度 / 考試篩選
- 呼叫搜尋 service
- 顯示搜尋結果

## 為什麼不用直接改 repository

這次需求是「搜尋結果排序」而不是「字庫基礎資料順序」。

若直接改 repository 或 asset 載入順序，會把搜尋語意與資料來源耦合在一起，之後要再調整排序規則會比較難維護，也可能影響其他使用 `_entries` 原始順序的頁面。

## 測試策略

新增 `WordBankSearchService` 的單元測試，至少覆蓋：

- 搜尋 `money` 時，`money` 會排在 `baldmoney`、`commoney` 前面
- 字首匹配會排在單字內包含前面
- 解釋文字命中會排在英文單字命中之後
- 空搜尋時維持既有限制筆數

## 未來可延伸方向

- 支援 typo tolerance / fuzzy search
- 依使用者加入或查看行為做個人化排序
- 對超短 query 加 debounce 或增量搜尋
- 針對中文搜尋再拆出更細的優先級
