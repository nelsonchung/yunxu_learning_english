# Word Bank Meaning Search

- Date: 2026-04-04
- Status: Draft

## 1. 背景

目前字庫搜尋頁的主要使用情境是輸入英文單字，從內建字庫中找到對應單字後加入複習庫。

但實際使用上，使用者也會先想到中文意思，例如輸入 `麵包`、`補償`、`中性`，希望反查對應英文單字。這代表字庫搜尋不應只被視為英文查英文，而應支援「中文查 meaning」。

## 2. 現況

### 2.1 現有程式行為

目前 `WordBankSearchService` 已有下列搜尋規則：

1. `word == query`
2. `word.startsWith(query)`
3. `word.contains(query)`
4. `meaning.contains(query)`

也就是說，從 service 邏輯來看，只要輸入的中文關鍵字有出現在 `meaning` 內，理論上已經可以命中結果。

### 2.2 目前缺口

雖然 service 已有基本能力，但整體功能還沒有被正式定義成「支援中文 meaning 搜尋」，主要缺口如下：

1. 搜尋框提示文字目前只展示英文例子，使用者不會知道可以輸入中文。
2. 測試案例只有英文搜尋與英文 meaning 命中，缺少中文查詢的驗證。
3. 搜尋正規化目前只有 `trim()` 與 `toLowerCase()`，對中文標點、全形空白、資料噪音的容錯不足。
4. `meaning` 命中目前只有單一 bucket，未針對中文搜尋品質做更細排序。

## 3. 目標

本次需求的目標如下：

1. 使用者可直接輸入中文關鍵字，查到 `meaning` 含有該關鍵字的單字。
2. 不破壞目前英文搜尋排序與程度篩選行為。
3. 第一階段不調整字庫 schema，不新增資料表，不重做 repository。
4. 讓這項能力在 UI、文件與測試層都成為正式支援功能。

## 4. 非目標

這一版先不處理下列項目：

1. 拼字容錯或 fuzzy search
2. 中文同義詞擴展
3. 斷詞或語意搜尋
4. 23 萬字級別的索引重構
5. `meaning` 資料內容品質的大規模清洗

## 5. 設計判斷

### 5.1 為什麼第一階段不改資料結構

目前 `BuiltinWordEntry` 已經有 `meaning` 欄位，而且 asset JSON 內容也已經提供中文意思。

因此要支援中文查詢，第一階段最合理的方式不是新增欄位或索引，而是把既有 `meaning contains` 行為正式化，補足：

1. 搜尋正規化
2. UI 提示
3. 測試覆蓋
4. 驗收標準

這樣改動最小，也最容易驗證風險。

### 5.2 為什麼不只靠「現在其實能查到」

目前雖然技術上已能命中部分中文查詢，但若沒有文件、測試與 UI 提示，這仍然只算隱性能力，不算正式功能。

當後續重構搜尋排序或 repository 時，這個行為很容易被不小心改壞，卻不會有人察覺。

## 6. 推薦方案

建議採兩階段方案。

### 6.1 Phase 1：正式支援中文 meaning 搜尋

這一階段以最小改動達成正式可用。

#### 搜尋邏輯

維持現有主結構，但補上查詢與資料正規化。

建議規則：

1. query 先做 `trim`
2. 英文仍做 `toLowerCase`
3. 將全形空白轉半形空白
4. 將連續空白壓成單一空白
5. 比對 `meaning` 時，可額外忽略常見中文標點差異

第一階段仍可維持原本 bucket 順序：

1. `word exact`
2. `word prefix`
3. `word contains`
4. `meaning contains`

理由是中文 query 幾乎只會命中 `meaning`，因此即使不新增語言判斷，也已足夠支援主要使用情境。

#### UI 文案

字庫搜尋頁需明確告知支援中文：

1. 搜尋框提示文字改為 `例如：co、trans、ability、麵包、補償`
2. 搜尋框下方說明補一句 `可搜尋英文單字或中文意思`
3. 空結果文案改為 `查不到符合的英文單字或中文意思，請換個關鍵字試試看。`

#### 測試

至少新增以下單元測試：

1. 中文 query 可命中 `meaning`
2. 中文 query 只命中 `meaning` 時，仍會回傳正確結果
3. 英文搜尋排序不受影響
4. 空白與全形空白不影響中文搜尋結果

### 6.2 Phase 2：提升中文搜尋排序品質

若第一階段上線後，發現中文命中結果很多且排序不夠精準，可再進一步拆 bucket：

1. `word exact`
2. `word prefix`
3. `word contains`
4. `meaning exact`
5. `meaning prefix`
6. `meaning contains`

這能讓搜尋 `補償` 時，`meaning` 恰好等於或開頭就是 `補償` 的單字排得更前面，而不是所有中文 meaning 命中都只依英文單字字母排序。

## 7. 實作範圍

### 7.1 需要改動的檔案

主要改動點預計如下：

1. `lib/domain/services/word_bank_search_service.dart`
   - 抽出搜尋正規化 helper
   - 視需求加入 meaning 專用比對策略

2. `lib/presentation/pages/word_bank_page.dart`
   - 更新搜尋框提示文字
   - 更新搜尋說明文案
   - 更新空狀態文案

3. `test/word_bank_search_service_test.dart`
   - 補中文 meaning 搜尋案例
   - 補正規化案例

### 7.2 不需要改動的檔案

第一階段不建議改動：

1. `lib/domain/models/builtin_word_entry.dart`
2. `lib/data/repositories/builtin_word_bank_repository.dart`
3. `assets/word_bank/*.json`

除非驗證後確認資料品質問題大到已影響搜尋正確率，否則不應把需求擴張成資料清洗專案。

## 8. 正規化策略

### 8.1 Query 正規化

建議新增 `_normalizeQuery()` 或類似 helper，至少處理：

1. 去除前後空白
2. 英文轉小寫
3. 將 `\u3000` 全形空白轉為一般空白
4. 將多個空白合併成一個

### 8.2 Searchable Text 正規化

對 `meaning` 可增加較寬鬆的比對版本，例如：

1. 保留原始文字版本
2. 另建立忽略常見標點的版本

常見可忽略字元例如：

- `，`
- `。`
- `；`
- `：`
- `、`
- `（）`
- `()`

第一階段不一定要完全移除所有標點，但至少要有一致的 helper，避免搜尋規則散落在頁面與 service。

## 9. 驗收標準

完成後應滿足以下條件：

1. 輸入 `麵包` 時，可找到 `naan` 或其他 `meaning` 含有麵包的單字。
2. 輸入 `中性` 時，可找到 `N` 或 `n` 等對應單字。
3. 輸入 `補償` 時，可找到 `meaning` 含有補償的單字。
4. 輸入英文 `money` 時，既有排序規則不變。
5. 程度過濾與 TOEIC 過濾仍正常生效。
6. 測試需明確覆蓋中文查詢案例。

## 10. 風險與注意事項

### 10.1 資料品質不一致

部分字庫內容可能有：

1. 中英混排
2. 多個義項以標點串接
3. 不一致空白
4. 極少數異常字串

因此搜尋正規化應以「提升容錯」為目標，而不是假設所有 `meaning` 都格式完全一致。

### 10.2 不要過早做語言判斷

第一階段不建議為 query 強制做「中文模式」或「英文模式」切換。

原因是：

1. 現有邏輯同時搜尋 `word` 與 `meaning` 已足夠
2. 強制語言切換會增加邊界情況，例如混合 query、符號 query
3. 現階段真正缺的是正規化、UI 與測試，而不是查詢模式判斷

### 10.3 不要把需求擴張成全文檢索專案

這次需求的核心是「讓中文 meaning 搜尋正式可用」，不是一次導入全文索引、斷詞、拼字校正或多語搜尋平台。

若後續字庫規模成長到必須重做查詢層，再與 `23 萬字庫重構方案` 一併處理即可。

## 11. 建議實作順序

建議依以下順序落地：

1. 補文件，確認需求邊界
2. 調整 `WordBankSearchService` 的正規化 helper
3. 補中文搜尋測試
4. 更新字庫搜尋頁文案
5. 用真資料手動驗證 `麵包`、`補償`、`中性`

## 12. 後續可延伸方向

若未來需要更好的中文搜尋體驗，可再評估：

1. `meaning exact / prefix / contains` 分級排序
2. 常見同義詞或別名映射
3. 搜尋 debounce
4. 大字庫下的索引化查詢
